CodeMate — Redesigned Architecture

A real-time collaborative code editor built on a server-authoritative Yjs CRDT, JWT-guarded WebSockets, and persisted, ownable CodeSpaces.

1Overview

CodeMate is a real-time collaborative code editor where any user signs in with Google, spins up a single-file CodeSpace, and shares a link so others can edit alongside them. The browser runs a React 19 + Vite app with a Monaco editor bound to a Yjs document; keystrokes flow as small binary CRDT updates over a WebSocket to a socketServer that holds the authoritative in-memory Y.Doc for each room and debounce-persists it to Postgres. A separate Express REST API handles login, CodeSpace CRUD, and code execution via Judge0, using Redis purely as a cross-process message bus. The whole stack is TypeScript across a pnpm + Turborepo monorepo.

2Service Topology

Three deployable services sit between the browser and shared infrastructure. The Express server (:3000) owns REST: auth, CodeSpace CRUD, and the run endpoint. The socketServer (ws :8080) owns realtime sync and is the Yjs authority. Redis is the cross-process bus — the Express server publishes run results, and the socketServer subscribes. Both services read and write Postgres/Neon. External dependencies are Google OAuth (login) and Judge0 (execution).

3Data Model

Three tables. A User owns many CodeSpace rows (the owner relation), and users relate to CodeSpaces many-to-many through the CodeSpaceCollaborator join table. The CodeSpace.ydocState column (encoded Yjs bytes) is the source of truth for document content; the password column is gone because login is Google-only.

erDiagram
  User ||--o{ CodeSpace : "owns"
  User ||--o{ CodeSpaceCollaborator : "joins"
  CodeSpace ||--o{ CodeSpaceCollaborator : "has"

  User {
    uuid id PK
    string googleId UK
    string email UK
    string name
    string avatarUrl "nullable"
  }

  CodeSpace {
    cuid id PK "link / room / doc key"
    uuid ownerId FK
    string title "default Untitled"
    int language "Judge0 language id"
    bytes ydocState "encoded Y.Doc — SOURCE OF TRUTH"
    bool allowEdit "default false — global switch"
    datetime createdAt
    datetime updatedAt
  }

  CodeSpaceCollaborator {
    cuid codeSpaceId PK_FK
    uuid userId PK_FK
    enum role "owner | editor | viewer"
    datetime lastOpened
  }

The collaborator row's composite primary key is (codeSpaceId, userId). In the MVP, the role column is only ever written as owner or viewer; it is the future home of real per-user roles and invites.

4Authentication Flow

Google is only the front door. CodeMate runs the standard OAuth Authorization-Code flow to identify the user, then upserts a User by googleId and mints its own access + refresh JWTs. From that point on, the app's JWTs are the session currency — Google is not contacted again. There are no passwords anywhere in the system.

  S-->>C: set tokens (app JWTs = session currency)
  Note over C,S: No passwords. Google never contacted again per request.

5Realtime Collaborative Editing (Yjs)

Editing is driven by Yjs, a CRDT library. In the browser, y-monaco binds the Monaco model to a shared Y.Doc. When a user types, Yjs emits a small binary update (not the whole file), which is sent over the WebSocket. The socketServer applies that update to the authoritative in-memory Y.Doc for the room and relays it to every other connected client. A new joiner gets the current state through the Yjs sync handshake (state-vector exchange), so it never has to replay history. The server debounce-persists the encoded doc state to CodeSpace.ydocState in Postgres.

Authority & affinity. The server is the authoritative holder of each room's Y.Doc. Room affinity pins every socket for a given codeSpaceId to one socketServer replica — a single master, with no multi-master merge. Presence and colored cursors ride on y-protocols/awareness, kept out of the persisted document.

6Socket Connect: Auth + Authorization

The socket is where permissions are enforced, not merely displayed. On connect, the client supplies its access token (query param or first AUTH message). The server verifies the JWT signature locally and extracts userId from the token — it never trusts a client-sent userId. It loads the CodeSpace, derives the role, and if the connection is read-only it silently drops inbound Yjs updates. The server is the bouncer; any client-side gating is cosmetic and untrusted.

7Run Code Path

Running code is a one-off HTTP request, deliberately kept out of the Yjs sync path. The client materializes the current code as plain text from the Yjs doc, attaches batch stdin, and POSTs to Express. Express calls Judge0, then publishes the result to the Redis channel room:{codeSpaceId}. The socketServer replica that holds that room is subscribed and fans stdout / stderr / compile output / time to everyone in the room — so all participants see the same run result. HTML/CSS CodeSpaces are run-disabled.

8CodeSpace Lifecycle

A user can create CodeSpaces up to a hard cap of 5 — the 6th create is rejected. The dashboard shows My CodeSpaces (owned) plus Recently opened (derived from CodeSpaceCollaborator rows). Opening a link /<codeSpaceId> joins as editor or viewer per the global Allow Edit switch and writes/updates a collaborator row so the space is remembered. Only the owner may delete: deletion kicks sockets, drops the in-memory doc, deletes the rows, and frees a slot.

9Key Departures from the Old Design

The redesign changes five load-bearing decisions:

Area Old design New design
CodeSpace identity Ephemeral, keyed on userId (one per user) Persisted, owned, many per user (cap 5)
Sync model Naive full-document broadcast Yjs CRDT with a server-authoritative Y.Doc
Socket trust Trusts client-sent userId Socket verifies the JWT and enforces permissions itself
Link identity Watch-ID == userId codeSpaceId is the link / room / doc key
Redis role Sync source-of-truth Demoted to cross-process bus; Postgres (ydocState) is source of truth
CodeMate — redesigned architecture. TypeScript · pnpm + Turborepo · React 19 + Vite + TailwindCSS + Monaco (Shadcn UI) · Express · native ws · Yjs (y-monaco, y-protocols/awareness) · Prisma + Postgres (Neon) · Redis pub/sub · Judge0 · Google OAuth + app-issued JWTs.
