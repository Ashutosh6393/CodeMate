problem
The wire protocol exists in three half-versions — two diverging unions plus an untyped Redis dialect — and every receiver does a raw JSON.parse.

solution
One @repo/protocol module: the full typed message set behind encode/decode, validated on the seam, imported by both client and socketServer.

  • leverage: one interface, every send/receive site
  • locality: protocol drift fixed once
  • interface shrinks; parsing absorbed inside
  • the interface is the protocol test surface

problem
A Live Share is a real domain concept, but its state (Realtime Code, Language, Allow Edit, Viewer set) has no module — it's raw Redis keys, createClient, and JSON smeared across six handlers.

solution
A LiveShareStore with a small interface hiding key construction, serialization, and the subscriber lifecycle. Real Redis in prod, in-memory fake in tests.

  • two adapters justify the seam: Redis vs in-memory
  • handlers become testable without Redis
  • locality: key + colon bugs concentrate
  • delete it → state logic reappears in 6 handlers
Preserves ADR-0003 (naive full-document broadcast) and ADR-0002 (Redis pub/sub). This restructures how state is accessed, not the broadcast model — no OT/CRDT introduced.

Authentication
register(creds) · authenticate(creds)
refresh(token) · verify(token)
accepts: userStore · hasher · tokens · clock

problem
Each auth controller mixes validation, DB, hashing, token minting, and cookies; prisma and argon2 are created inline, so there's nothing to test but the whole HTTP round-trip.

solution
An Authentication module that returns tokens-or-error and accepts its user store, hasher, token signer, and clock. Controllers shrink to HTTP adapters; jwtHelpers and response are absorbed.

  • accept dependencies → real prisma vs in-memory store
  • return results → assert tokens, not cookies
  • interface shrinks; two shallow wrappers absorbed
  • cookie maxAge drift concentrates in one place

problem
Run Code is welded to the controller: axios options, the judge0-ce endpoint, and the env key are all inline, so the submit path can't be tested without a live third party.

solution
A CodeRunner interface — run(code, language, stdin) → RunResult — with a Judge0 adapter behind it and a fake for tests.

  • two adapters: Judge0 in prod, fake in tests
  • submit path testable offline
  • provider swap = one adapter
Aligns with ADR-0004, which already names Judge0 lock-in as the cost. A port doesn't reverse that decision — it isolates the lock-in to one adapter. Side note (bug, not architecture): /submitcode has no auth middleware applied (routes/index.ts:14) despite authCheck existing.

problem
The one Redis publisher is exported from index.ts, re-exported from a controller, and imported back by utils — a cycle, with two import paths for one thing.

solution
This mostly dissolves once Candidate 2 lands: the publisher and key helpers move behind LiveShareStore, leaving a single import site.

  • removes the import cycle
  • folds into Candidate 2 — not standalone work
top recommendation

Start with Candidate 2 — the LiveShareStore

It's the deepest module hiding the most behaviour, it's where the real collaboration bugs live (inconsistent keys, lost serialization, subscriber lifecycle), and it has two genuine adapters — Redis in prod, in-memory in tests — so it makes the entire socketServer testable for the first time. It also dissolves Candidate 5. Pair it with Candidate 1 (@repo/protocol): together they give the real-time path one interface for the wire and one interface for the state.
